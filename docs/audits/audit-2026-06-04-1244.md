## 1. Overall Assessment

**Readability**: B+ (7.5/10)  
**Maintainability**: B+ (7.5/10)

The codebase is **well-engineered** with strong documentation discipline,
consistent architectural patterns, and active technical debt management. It
shows deliberate evolution through 24 ADRs and a carefully curated TODO
pipeline.

The main readability drag is **function length in the UI layer** — several
functions exceed 200 lines with deeply nested conditionals. The main
maintainability drag is **minor duplication** in the serialization layer and
**variable docs/src quality** across modules.

## 2. Strengths & Patterns Worth Preserving

### 2.1 Docstring Excellence

Every public item has a thorough docstring. The quality is consistently high:

- **Parameters** are individually documented with types and semantics.
- **Return values** describe what the caller receives.
- **`# Panics`** documents every invariant-violation panic with the condition.
- **`# Errors`** documents every `Result` failure mode.
- **Side effects** are called out explicitly.

Example from `pixel.rs`:

```
/// Convert a straight-alpha color to premultiplied alpha.
///
/// Uses fixed-point arithmetic `(value * alpha + 128) * 257 >> 16` for correct
/// rounding. Fully opaque colors pass through unchanged; fully transparent
/// colors become `Color32::TRANSPARENT`.
///
/// **Caller must supply straight (non-premultiplied) RGB.**
/// Calling this on an already-premultiplied pixel will darken colors again.
///
/// # Parameters
///
/// * `color` — Straight-alpha color to convert to premultiplied format.
```

This is the gold standard. Every module follows it.

### 2.2 Uniform Tool Pattern

Every drawing tool is a **free function returning `UndoRecord`**:

```rust
// Pattern — all tools follow this signature shape
pub fn draw_square(...) -> UndoRecord
pub fn draw_circle(...) -> UndoRecord
pub fn draw_bucket_fill(...) -> UndoRecord
pub fn draw_square_line(params: BrushStrokeParams, radius: u32) -> UndoRecord
pub fn draw_circle_line(params: BrushStrokeParams, radius: u32) -> UndoRecord
```

This makes tools **testable in isolation**, **composable**, and **predictable**.
Adding a new tool means adding one function and wiring it in two places
(center.rs dispatch and left.rs UI).

### 2.3 Builder Pattern for Brush Parameters

`BrushStrokeParams` uses a builder pattern to reduce 11-parameter function
signatures to readable chains:

```rust
BrushStrokeParams::builder(canvas, color, layer, visited, stamp, drag_processed, ds_val)
    .start(x1, y1)
    .end(x2, y2)
    .alpha_overlay(true)
    .build()
```

The builder captures invariant fields once, then lets callers vary only
start/end/alpha per line segment.

### 2.4 LayerAction Deferred-Enum Pattern (right.rs)

The `LayerAction` enum in `right.rs` is a clever pattern for resolving borrow
conflicts with egui's immediate-mode UI:

```rust
enum LayerAction {
    Delete(usize),
    MoveUp(usize),
    MoveDown(usize),
    Select(usize),
    ToggleVisible(usize),
    Rename(usize, String),
    SetMode(usize, LayerMode),
}
```

Widgets collect `LayerAction` values during egui iteration (which borrows the
layer list immutably). After the loop, actions are applied with mutable access
to the document. This pattern should be reused if new UI panels need to mutate
layers.

### 2.5 Per-Row Span Deduplication in Square Brush

`stamp_line_positions` in `square_brush.rs` uses per-row `row_min_x`/`row_max_x`
tracking to avoid re-stamping pixels already covered by overlapping brush
footprints. This is an O(L·R) optimization over the naive O(L·R²) approach.

### 2.6 Async File IO via mpsc

Each IO manager owns its own mpsc channel pair:

```
DialogManager  ──mpsc──►  poll_dialog_results()
SaveManager    ──mpsc──►  poll_save_results()
ExportManager  ──mpsc──►  poll_export_results()
LoadImportManager──mpsc──►  poll_load_import_results()
```

The frame loop explicitly orchestrates them via `poll_file_results()`. This
avoids shared-state concurrency — no locks, no `Arc<Mutex>`, no
`async`/`await` runtime.

### 2.7 RenderState Machine

Three-state render control (`ActiveWake`/`IdleThrottled`/`UnfocusedFrozen`) with
duration-based transitions. Clean design:

```
UnfocusedFrozen ──► IdleThrottled (on focus)
IdleThrottled   ──► ActiveWake(550ms) (on interaction)
ActiveWake(dur) ──► IdleThrottled (when duration expires)
```

### 2.8 ADR Discipline

24 ADRs with consistent front matter, context, decision, rationale, alternatives,
and impact sections. Superseded ADRs are cross-referenced (`Superseded-By:
ADR-0024`). This is rare and valuable.

### 2.9 TODO Pipeline Discipline

- Items prioritized P0–P5, change-level B0–B5.
- Each item tagged with the commit hash of when it was created.
- Archived items record why they were denied/implemented with reasoning.
- Empty sections preserved as placeholders (intentional gap signaling).

### 3.3 Duplicate Validation in Serialization

`read_canvas` (zstd) and `read_canvas_xz` (xz2) are structurally identical —
only the decoder differs. Example of the duplication pattern:

```rust
// src/files.rs:46-87
fn read_canvas(reader: impl Read) -> anyhow::Result<Canvas> {
    let decoder = zstd::Decoder::new(reader)?;    // ← only difference
    let limited = decoder.take(MAX_DECOMPRESSED_BYTES);
    let mut canvas: Canvas = serde_json::from_reader(limited)...;
    // 20 lines of validation...
}

// src/files.rs:229-270
fn read_canvas_xz(reader: impl Read) -> anyhow::Result<Canvas> {
    let decoder = xz2::read::XzDecoder::new(reader); // ← only difference
    let limited = decoder.take(MAX_DECOMPRESSED_BYTES);
    let mut canvas: Canvas = serde_json::from_reader(limited)...;
    // identical 20 lines of validation...
}
```

Same duplication exists for `write_canvas` / `write_canvas_xz` and their
`save_to_path` / `load_from_path` wrappers. Adding a new compression format
(e.g., zlib, LZ4) would require duplicating the pattern a third time.

**Fix**: Accept a generic decoder factory: `fn read_canvas_generic(reader: impl
Read, decoder: impl Fn(impl Read) -> impl Read) -> ...`.

### 3.4 Docstring Duplication (undo.rs)

`UndoRecord::maybe_snapshot` has its full docstring duplicated verbatim — the
same 30-line block appears at lines 170-208 and again at 210-248. Both blocks
are identical including the `///` markers. This is a copy-paste error.

### 3.5 Builder Field Shadowing (brush_params.rs)

The `BrushStrokeParamsBuilder` has fields with the same names as
`BrushStrokeParams`, but when debugging, the builder and struct are visually
identical:

```rust
pub struct BrushStrokeParams<'a> {
    pub start_x: u32, pub start_y: u32, pub end_x: u32, pub end_y: u32,
    pub canvas: &'a mut Canvas, pub color: Color32, pub layer: usize,
    pub visited: &'a mut [u32], pub stamp: u32, pub alpha_overlay: bool,
    pub drag_processed: &'a mut [u32], pub drag_stamp_value: u32,
}

pub(crate) struct BrushStrokeParamsBuilder<'a> {
    // Same 12 fields, same names, same types
    start_x: u32, start_y: u32, end_x: u32, end_y: u32,
    // ... all 12 again
}
```

The naming implies they're the same thing, but one is a builder and the other
is the finished product. Consider naming the builder `BrushStrokeBuilder` or
making it a private inner type of `BrushStrokeParams`.

### 3.6 Match Arm Uniformity

The `CurrentTool` enum is matched in many places with exhaustive arms:

- `center.rs`: brush preview (8 arms: Circle, Eraser(Circle), Square,
  Eraser(Square), BucketFill, Eyedropper, Pan, Stamp, CustomBrush)
- `center.rs`: `apply_stroke` (9 arms including eraser sub-variants)
- `left.rs`: UI rendering (all variants)

When a new tool is added, every exhaustive match must be updated. The compiler
will catch this because `CurrentTool` is a non-exhaustive enum pattern — but
each match adds cognitive load.

`docs/src/pixel.md` references `blend_pixel_range` which was merged into
`apply_single_layer`. Noted in TODO.md but the doc is stale.

## 4. Maintainability Analysis

### 4.1 MyApp State Centralization

`MyApp` owns all application state as direct fields (11 fields):

```
MyApp {
    document: Document,
    tool_configuration: ToolConfiguration,
    undo: UndoHistory,
    dialog_manager: DialogManager,
    save_manager: SaveManager,
    export_manager: ExportManager,
    load_import_manager: LoadImportManager,
    ui: UIState,              // 25 sub-fields
    gpu_texture: Option<GpuTexture>,
    stamp_library: Library<StampEntry>,
    brush_library: Library<BrushEntry>,
}
```

Plus all 7 UI panels, frame lifecycle, and dialog methods are `impl MyApp`
blocks in separate files. This means:

- **Any panel can access any state**. There's no access control — `show_top_panel`
  can modify the stamp library, brush preview, document, undo history, etc.
- **Testing a single panel requires constructing full `MyApp`** (see
  `tests/common.rs` `create_test_app` which creates all 11 fields).
- **Adding a feature** means adding fields to `UIState` or `MyApp`, which grows
  the implicit dependency graph.

This is inherent to egui's pattern, but some mitigation is possible: split
state into focused sub-structs passed to panel functions rather than giving
panels access to the entire `MyApp`.

### 4.2 AssetEntry Trait Verbosity

The `AssetEntry` trait requires 11 methods for implementation:

```rust
pub trait AssetEntry: Sized {
    fn name(&self) -> &str;
    fn filename(&self) -> &str;
    fn filename_mut(&mut self) -> &mut String;
    fn pixels(&self) -> &[Color32];
    fn width(&self) -> u32;
    fn height(&self) -> u32;
    fn texture_handle(&self) -> &Option<TextureHandle>;
    fn texture_handle_mut(&mut self) -> &mut Option<TextureHandle>;
    fn dir_name() -> &'static str;
    fn json_field_name() -> &'static str;
    fn extra_index_fields(&self) -> Vec<(&'static str, serde_json::Value)>;
    fn from_parts(name, filename, pixels, w, h, extra) -> Self;
}
```

Both `BrushEntry` and `StampEntry` implement this with near-identical
boilerplate. Many methods are trivial accessors that could have default
implementations or be generated by a derive macro.

### 4.3 Config Save on Every Autosave Tick

`handle_config_save` writes `config.json` on the same 2-minute cadence as
autosave, regardless of whether any config state changed:

```rust
pub(crate) fn handle_config_save(&mut self) {
    if self.ui.time_elapsed.saturating_sub(self.ui.last_autosave_time)
        >= Duration::from_mins(AUTOSAVE_INTERVAL_MINUTES)
    {
        self.save_config();  // writes even if nothing changed
    }
}
```

This causes unnecessary disk writes on every tick. Add a dirty flag
(`config_dirty: bool`) that is set only when a persisted field actually
changes.

### 4.4 Feature-Gate Sprawl

`#[cfg(test)]` annotations are scattered throughout source files:

| Module             | `#[cfg(test)]` items                                                 |
| ------------------ | -------------------------------------------------------------------- |
| `undo_history.rs`  | 9 methods (accessors, setters)                                       |
| `asset_library.rs` | 3 methods, 1 type                                                    |
| `canvas.rs`        | 1 method (`DirtyRectList::clear()`)                                  |
| `files.rs`         | 4 functions (`save_canvas_to_bytes`, `load_canvas_from_bytes`, etc.) |
| `undo.rs`          | 1 constant (`RLE_SHORT_RUN_THRESHOLD` is `pub(crate)`)               |

This makes it harder to scan the "release" code path because test-only code is
interleaved. Consider:

- Moving test helpers into `src/tests/` where practical.
- Grouping `#[cfg(test)]` blocks at the bottom of each file.
- Using `#[cfg(test)] pub(crate) fn` accessors instead of `pub fn`.

### 4.5 ClippedOverlap suppress_base Correctness

The clipping logic in `pixel.rs` lines 803-811:

```rust
let mut suppress_base = vec![false; layers.len()];
for (i, info) in layers.iter().enumerate() {
    if info.mode == LayerMode::ClippedOverlap {
        suppress_base[base_indices[i]] = true;
    }
}
```

For clip chains deeper than 2 levels (e.g., A:Normal -> B:ClippedOverlap ->
C:ClippedOverlap), this sets `suppress_base[A] = true` for B, then
`suppress_base[A] = true` again for C. But the base index resolution in
`compute_base_indices` walks upward — for C:ClippedOverlap, `current_base` was
already set to B:ClippedOverlap (because it's not Clipped), so C's base
index is B, not A. This means:

- `suppress_base[B] = true` (for C), but B is ClippedOverlap, not Normal
- A:Normal's `suppress_base` was only set by B, not C
- The iteration in `blend_layers` may skip the wrong layers

This is already noted as TODO item (P3) and needs a test with 3+ layer clip chains.

### 4.6 Undo Snapshot Size Limit Gap

`files.rs` uses `MAX_DECOMPRESSED_BYTES` (512 MB) for file loading, but
`undo.rs:maybe_snapshot` decompresses full-layer snapshots without a size
limit:

```rust
// undo.rs:267-269
let json = serde_json::to_vec(&before_layer)?;
*full_layer_before = Some(zstd::encode_all(std::io::Cursor::new(json), level)?);
```

The `serde_json::to_vec` allocates a full uncompressed buffer without
constraints. Noted in TODO.md as a security issue (P2).

### 4.7 Drag Accumulator Split Edge Case

When a single drag gesture exceeds `MAX_DRAG_FRAMES` (5000), the accumulator
finalizes mid-gesture and starts a new one. The merge logic in
`finalize_drag_accumulator` checks if the last stroke stack entry matches the
same drag and merges. But if multiple `MAX_DRAG_FRAMES` splits occur, only the
_last_ prior record is merged — all earlier split records remain adjacent but
separate. This means undoing a very long drag stroke requires N undo steps
instead of 1.

### 4.8 ICO Format Scale-Down Allocation

For ICO export with images >256×256, `export_as_image` allocates an
intermediate `RgbaImage`, runs Lanczos3 resampling, then extracts raw bytes:

```rust
// files.rs:632-670
let img = image::RgbaImage::from_raw(width, height, raw_output).expect(...);
let img = image::imageops::resize(&img, new_width, new_height, Lanczos3);
let ico_raw = img.into_raw();
```

This is one of the few places where `raw_output` is wrapped in an `RgbaImage`.
The resize could operate directly on the byte buffer. Not a performance issue
at typical icon sizes, but structurally inconsistent with the rest of
`export_as_image` which avoids `RgbaImage`.

## 5. Module-by-Module Observations

### 5.1 `src/main.rs` (96 lines)

**Good**: Minimal entry point. Only does allocator setup, directory creation,
and eframe launch. No business logic.

**Observation**: The `use` statement for `wgpu` is behind a function-local
`use` to avoid importing at module level. Clean approach.

### 5.2 `src/app/mod.rs` (777 lines)

**Good**:

- Clear separation of identity constants, performance constants, types, and
  `MyApp` construction.
- `estimate_canvas_memory` is a well-documented utility.
- `EXPORT_FORMATS` table is self-documenting.

**Issues**:

- `MyApp::new()` is 137 lines with multiple concerns (directory setup, GPU
  texture, config loading, library loading).
- `UIState` has 25 fields — some are transient (cursor position), some are
  persistent (zoom, pan). Mixing persist/transient in one struct makes config save
  harder.
- Dialog state (`DialogState`, `ErrorState`, `ToastState`, `ProgressState`) are
  split across 4 structs when they could be a single `UiOverlayState`.

**Pattern note**: `GpuTexture` holds `queue: Arc<wgpu::Queue>` which is cloned
from the render state. The `Arc` is necessary because `wgpu::Queue` is not
`Send` but `recreate_gpu_texture` passes it to a lambda. This is a subtle
concurrency constraint that deserves a comment explaining _why_ `Arc` is
needed.

### 5.3 `src/app/frame.rs` (395 lines)

**Good**:

- `poll_file_results` is logically organized with numbered comments for each
  of the 7 steps.
- `update_render_state` is clean — the render state machine is easy to follow.
- `recreate_gpu_texture` has good error handling for oversized canvases.

**Issues**:

- `poll_file_results` is 168 lines with 7 sequential concerns. Each concern
  (dialog dispatch, save results, deferred action, load/import, progress
  tracking, stamp/brush transfer, recent files) could be extracted.
- Progress tracking (lines 106-137) uses a priority chain: load > import >
  export > save > idle. The `was_saving` capture before polling (line 29) is a
  clever pattern but fragile — if any poll function changes `ui.progress`, the
  capture becomes stale.

### 5.4 `src/document.rs` (631 lines)

**Excellent**. Clean single-responsibility with well-factored methods.

- Layer operations (add, delete, move up/down, select, toggle visibility,
  opacity, mode, rename) are each their own focused method.
- `UndoRecord` push happens in the same method as the mutation — atomic.
- `canvas_mut()` provides COW semantics with a clear docstring.

**Observation**: The `ModifyLayer` undo record captures old _and_ new values
for every field, even when only one field changed. E.g., `set_layer_opacity`
captures `old_visible`, `old_name`, `old_mode` even though those didn't change.
This means `ModifyLayer` records are larger than necessary for single-property
changes. A per-field undo scheme would be more memory-efficient but adds
complexity. Current approach is simpler and correct.

### 5.5 `src/canvas.rs` (523 lines)

**Excellent**. Clean types with serde support and Debug implementations that
skip pixel data (key for performance).

- `LayerMode` is well-documented with real-world analogies (Photoshop clipping
  masks, Inkscape clip groups).
- `DirtyRect` / `DirtyRectList` is a well-designed incremental tracking system
  with proximity-based merging and a maximum count threshold.

**Issue**: `rects_overlap_or_touch` expands `a` by `proximity` in all
directions, which means rects that are within 16px of each other get merged.
But this also means a rect on the right edge of the canvas could merge with
nothing (the expansion saturating is handled). Fine for correctness but worth
noting in the docstring.

### 5.6 `src/pixel.rs` (1397 lines)

**Excellent**. The SIMD blend pipeline is well-documented with clear algorithm
descriptions. The three-phase approach (scalar head, SIMD body, scalar tail)
is explained.

- `read_accumulator` / `write_accumulator` are clean abstractions for SIMD
  register access.
- `apply_single_layer` handles all four `LayerMode` variants with consistent
  parallelism logic.
- `blend_layers` and `blend_region` are documented with performance semantics.
- Tests are thorough with edge cases for all layer modes.

**Issue**: `compute_base_indices` and `suppress_base` are recomputed in both
`blend_layers` (full canvas) and `blend_region` (dirty rect). For `blend_region`
called per-row by `draw_checkerboard`, this recomputation happens every row.
For canvases with many layers, this is O(layers × rows) instead of O(layers × 1).
The TODO already notes this was considered and deemed ~100ns overhead —
acceptable.

**Observation**: `draw_checkerboard` processes 4-pixel batches with an early
exit when all 4 pixels are opaque. This is a good SIMD-friendly optimization.
But it still scans every pixel column-by-column (albeit in 4-wide batches).
For a 2000×1500 canvas with mostly opaque content, this is 3M checks per
frame. An optimization: skip the entire row if the corresponding output_rgba
row has no semi-transparent pixels.

### 5.7 `src/undo.rs` (576 lines)

**Good**. Run-length compressed per-pixel undo with three compression levels:

1. RLE uniform-run detection (`BeforePixels::All` for 8+ identical pixels).
2. zstd compression of `before_pixels` buffer for storage.
3. Optional full-layer zstd snapshot for strokes covering >50% of a layer.

**Issue**: The `maybe_snapshot` docstring is duplicated (lines 170-248).

**Issue**: `compress_and_store` does a double-pass on uniform runs — first
`slice.iter().all(|&p| p == slice[0])` to check uniformity, then
`extend_from_slice(slice)` if non-uniform. The uniform check reads the data,
then the non-uniform path reads it again. Noted in TODO (P4) but worth
fixing: combine the check and copy into a single pass.

### 5.8 `src/undo_history.rs` (663 lines)

**Excellent**. Well-factored undo/redo stack with drag accumulator.

- `DragFrame` / `DragAccumulator` cleanly separate per-frame data from
  accumulated state.
- `merge_frames` reverses frame order so overlapping paint undoes correctly.
- `push_undo_snapshot` provides the optional full-layer snapshot path.
- `scratch_buffers()` returns all three scratch buffers in one call to avoid
  borrow-checker fights — pragmatic.

**Observation**: `COMPRESSION_LEVEL: i32 = -1` is zstd's "default" level
(which is 3), not "fast" mode. The docstring says "fast mode" but `-1` is the
same as not specifying a level — it uses the library default (3, which is
medium speed/compression). Level `1` would be faster. Level `0` is no
compression. If the intent is fast compression, use level 1.

### 5.9 `src/files.rs` (791 lines)

**Mixed**: The serialization code is clean (streaming zstd/xz, no
intermediate allocations), but `export_as_image` at 252 lines is too long and
mixes concerns.

- The streaming pattern (`serde_json::to_writer` → zstd encoder → `File`) is
  excellent — zero intermediate allocations for the JSON blob.
- `export_as_image` handles 13 formats with 5 special cases (JPEG, HDR,
  Farbfeld, ICO, GIF) + a macro `export_via!` + a fallback match for AVIF,
  PNG, WebP, TIFF, TGA, PNM, QOI, EXR. This is a candidate for the Strategy
  pattern.

**Observation**: `MAX_IMPORT_DIMENSION` (16384) matches the GPU texture limit
set in `main.rs`. Intentional and well-documented.

**Observation**: The `ExportStrategy` trait is defined but only has one
implementation (`DefaultExportStrategy`). If only one strategy exists, the
trait adds indirection without benefit. Consider whether this level of
abstraction is needed, or inline the logic.

### 5.10 `src/ui/center.rs` (1365 lines)

**Observation**: `add_entry()` saves the PNG file, then calls `save_index()`.

## 8. Cross-Cutting Concerns

### 8.1 Error Handling Philosophy

The codebase has a documented error handling philosophy:

- **Panic** on invariant violations (logic bugs) — documented with `# Panics`.
- **`Result`** for recoverable errors (IO, deserialization, dialogs) —
  documented with `# Errors`.
- **`.expect()`** for "should never happen" scenarios.

In practice, the `.expect()` calls are the weakest link:

| Call                                                         | Justification       | Could be Result?          |
| ------------------------------------------------------------ | ------------------- | ------------------------- |
| `main.rs:62` — `ProjectDirs::from().expect()`                | No home directory   | Unlikely to recover       |
| `main.rs:66` — `create_dir_all().expect()`                   | FS permission error | Could propagate to caller |
| `main.rs:570` — `create_dir_all().expect()`                  | Same pattern        | Could propagate           |
| `main.rs:572` — `create_dir_all().expect()`                  | Same pattern        | Could propagate           |
| `undo_history.rs:124` — `compress_before().expect()`         | Compression failure | ⚠️ **Should be Result**   |
| `undo_history.rs:162-165` — `maybe_snapshot/compress_before` | Compression failure | ⚠️ **Should be Result**   |
| `undo.rs:331-339` — zstd decoder/read                        | Corrupt data        | ⚠️ **Could propagate**    |
| `undo.rs:384-389` — zstd decoder/from_reader                 | Corrupt data        | Already `expect`ing       |

The `push_undo`/`push_undo_snapshot` methods panic on compression failure,
which could be triggered by disk-full during large strokes. Noted in TODO.md
(P2).

### 8.2 Integer Overflow Awareness

The codebase uses `overflow-checks = true` in both dev and release profiles.
Key places where overflow matters:

- **Canvas dimensions**: `(width as usize).checked_mul(height as usize)` in
  `Canvas::new` — explicit overflow check with panic.
- **DirtyRect operations**: `saturating_sub`/`saturating_add` in
  `rects_overlap_or_touch` — safe.
- **Stamped line positions**: `saturating_sub`/`saturating_add` in
  `stamp_line_positions` — safe.
- **Visited stamp**: `wrapping_add` with explicit reset at 0 — correct.
- **Drag stamp**: Same pattern — correct.
- **Memory estimation**: Uses `u64` for pixel count — safe for large canvases.

### 8.3 SIMD Safety

All SIMD operations are through `wide::u32x4` and `bytemuck::cast_mut` — safe
abstractions. The safety invariants are documented:

```rust
// SAFETY: i < len, i + 4 <= len, and the slice is 4-element aligned.
let chunk: &mut [Color32; 4] = (&mut span[i..i + 4]).try_into().unwrap();
```

The `alpha_blend_simd_four` function uses `bytemuck::cast_mut` which has
runtime alignment checks in debug builds — documented in the `# Panics` section.

### 8.4 Premultiplied Alpha Invariant

All internal pixel data uses premultiplied alpha. This is documented in the
module docstrings and function signatures. The invariant:

- Input from color picker → `premultiply()` converts straight→premul.
- Output to image file → `unpremultiply()` converts premul→straight.
- All blend operations assume premultiplied inputs.
- `Color32::TRANSPARENT` (0,0,0,0) is correctly premultiplied.

This invariant is maintained consistently throughout the codebase.

### 8.5 Thread Safety

The codebase avoids shared-state concurrency:

- No `Mutex`, `RwLock`, or `Atomic` types.
- `mpsc` channels for all cross-thread communication.
- `Arc<Vec<u8>>` for clone-on-write canvas buffer sharing.
- `Arc<wgpu::Queue>` for GPU access (cloned, not shared-mutable).
- `rayon::par_iter` for data-parallel blend — functional, no shared mutation.

### 10.2 Coupling Analysis

| Pattern              | Coupling | Notes                                         |
| -------------------- | -------- | --------------------------------------------- |
| MyApp as god object  | High     | All panels access all state                   |
| Tool functions       | Low      | Pure-ish, take Canvas ref + return UndoRecord |
| File IO managers     | Low      | Isolated behind mpsc channels                 |
| UI panels → MyApp    | High     | Every panel method is `impl MyApp`            |
| Undo system → Canvas | Medium   | `undo_apply` / `redo_apply` mutate Canvas     |

The tightest coupling is between UI panels and `MyApp` — each panel method
receives `&mut self` (the full app) and can modify any state. This is the
egui pattern but would benefit from passing focused sub-structs.


Could become:

```rust
pub struct UIState {
```

## 12. Conclusion

SplatterIron is a **well-engineered project** that demonstrates:

- **Strong documentation discipline** — every public item documented, every
  ADR structured.
- **Consistent patterns** — tool pattern, builder pattern, async IO pattern.
- **Active maintenance** — TODO pipeline, ADR evolution, bug tracking.
- **Performance awareness** — SIMD, rayon, zstd, dirty-rect tracking, LTO.
- **Correctness culture** — overflow checks, unwrap_used lint, premultiplied
  alpha invariant.

The primary concerns are:

1. **Function length** in the UI layer, particularly `handle_canvas_interaction`
   (530 lines) and `apply_stroke` (277 lines with 4× boilerplate).
2. **Minor duplication** in the serialization layer (read_canvas/read_canvas_xz)
   and UI layer (left.rs stamp/brush gallery).
3. **Variable docs/src quality** — some files are exemplary (document.md),
   others undersized (frame.md, panels.md).
4. **Test depth** — 28 modules but some test only trivial properties.
5. **.expect() hygiene** — ~20 calls, some should propagate `Result`.

None of these are architectural problems. The codebase is healthy and
well-positioned for incremental improvement. The existing TODO pipeline
already tracks several of these items, confirming that the maintainers are
aware of the technical debt and have a strategy for addressing it.
